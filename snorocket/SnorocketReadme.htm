Snorocket ${project.version}

A fast Java implementation of the Dresden algorithm for classifying SNOMED CT.

Features:

  • Input File Formats
    • SNOMED CT Distribution format for Concepts and Relationships
    • KRSS (restricted to the description logic EL+bottom)
      This means for all, disjunction, negation, and instances are not supported.
    • native textual syntax
  • Output File Formats
    • SNOMED CT Distribution format for Relationships
    • state-snapshot file to support incremental classification
  • Interface
    • command line

Requirements:

  • Java 1.5 or later
  • Windows XP Pro
  • 2GB RAM
  • 100Mb free disk space

Installation:

Unzip the supplied zip file wherever you want it installed. The batch file can be used to invoke the classifier.

Usage:

snorocket
  (--conceptsFile filename --relationshipsFile filename|--krssFile filename|--snorocketFile filename)
  [--outputFile filename]
  [--inputState filename] [--outputState filename]
  [--noHeader] [--hideEquivalents] [--X]

Brackets and the vertical bar indicate mutually exclusive alternative arguments. Arguments inside square brackets are optional.

  • --conceptsFile filename --relationshipsFile filename

    Specifies the input files in SNOMED CT Distribution format for Concepts and Relationships.

  • --krssFile filename

    Specifies the input file in KRSS syntax.

  • --snorocketFile filename

    Specifies the input file in the native syntax.

  • --outputFile filename

    filename should be the path to a text file to write the classification results conforming to SNOMED CT Distribution format for Relationships.

    If filename is a single dash (-) then the output is written to the console.

  • --inputState filename

    To perform incremental classification, the initial base state needs to be established. filename should be the path to a file containing this base state.

  • --outputState filename

    To perform a subsequent incremental classification, the initial base state needs to have been stored. filename should be the path to a file in which to store the base state.

  • --noHeader

    The first line of the SNOMED CT Distribution format normally consists of text labels for each column and thus the input parser skips the first line. If, however, this header line is missing then this flag should be used to tell the parser not to skip the first line.

  • --hideEquivalents

    Normally SNOMED CT-based ontologies should not contain equivalent concepts. Thus, if equivalent concepts are discovered during classification, then they are printed out to the console. Specifying this flag will suppress this output.

  • --X

    Displays a list of non-standard options that are subject to change without notice.

Example:

snorocket --conceptsFile ontologies\almrf_concepts_stated.txt --relationshipsFile ontologies\almrf_relationships_stated.txt --outputFile -

Native textual syntax:

The syntax is specified formally below in Backus-Naur format.

Informally,

  • the syntax ignores blank lines and comment lines (those that start with a #)
  • general concept inclusions (GCIs) use
    • [ and ] for subsumption and ][ for equivalence,
    • + for conjunction,
    • and . is used to connect a role with a concept for existential restriction, while
  • role inclusions (RIs) use < for subsumption.

Examples
A ⊓ B ⊑ C    A + B [ C
A ⊓ B ≡ CA + B ][ C
∃r.A ⊓ Br.A [ B
r ⊑ sr < s
r ∘ s ⊑ tr s < t

BNF

<ontology> ::= { <statement> }
<statement> ::= <comment> | <gci> | <ri>
<comment> ::= "#" <text> <EOL>
<gci> ::= <expression> <inclusion> <expression>
<ri> ::= <role> {<role>} "<" <role>
<expression> ::=
               ( "(" <expression> ")"
               | role "." ( <concept> | "(" <expression> ")" )
               | <concept>
               ) <opt-conjunction>
<opt-conjunction> ::= "+" <expression> | ""
<inclusion> ::= "[" | "][" | "]"
<concept> ::= "top" | "138875005" | "bottom" | <identifier>
<role> ::= <identifier>
